Nine TLA+ files were added in `specs/tla/` covering three subsystems (Registry, SWIM, Supervisor). This review identifies correctness issues, weak invariants, redundancies, and naming inconsistencies.

## Opportunity Details

### 1. `IncarnationMonotonicity` is trivially true
- **What**: Current formula `incarnation[n][m] >= 0` is always true for naturals. Replace with action property: `[][incarnation'[n][m] >= incarnation[n][m]]_vars`
- **Where**: `RebarSwim.tla` line 192-194, `RebarSwim.cfg` (move from INVARIANT to PROPERTY)
- **Why**: The comment says "incarnation never decreases" but the formula doesn't check that. Bug: out-of-order gossip could overwrite higher incarnation — current invariant would miss it.

### 2. `StaleExitSafety` is trivially true
- **What**: Current formula `child_status[i] \in {"running", "stopping", "stopped"}` is always true by construction. Replace with action property: stale-PID exit (`child_pid[e.index] # e.pid`) must leave `child_pid` and `child_status` unchanged.
- **Where**: `RebarSupervisor.tla` line 287-290, `MCRebarSupervisor.cfg` (move from INVARIANT to PROPERTY)
- **Why**: The invariant is supposed to verify stale exits are no-ops but currently tests nothing meaningful.

### 3. `TombstoneBeforeAdd` duplicates `NoTombstoneResurrection`
- **What**: Both are identical (`e.tag \notin tombstones[n]`). Remove `TombstoneBeforeAdd` and its cfg reference — the comment already notes they're equivalent.
- **Where**: `RebarRegistry.tla` lines 184-190, `RebarRegistry.cfg`
- **Why**: Redundant invariants clutter the spec and waste TLC checking time.

### 4. Config file naming inconsistency
- **What**: Rename `MCRebarSupervisor.cfg` → `RebarSupervisor.cfg` to match Registry/Swim convention. Update run comment in spec.
- **Where**: `specs/tla/MCRebarSupervisor.cfg`, `RebarSupervisor.tla` header comment
- **Why**: Inconsistent naming creates confusion about which cfg file to use.

### 5. Unused `TagSymmetry`
- **What**: `TagSymmetry == Permutations(Tags)` is defined but intentionally unused (tag identity matters for merge). Add comment explaining why, or remove.
- **Where**: `MCRebarRegistry.tla` line 9
- **Why**: Without a comment, it looks like a bug (defined but not referenced).

### 6. `NoDoubleLiveChild` too weak
- **What**: Add `NoPIDSharing`: no two children share the same non-None PID. Current invariant only checks running→has PID.
- **Where**: `RebarSupervisor.tla` invariants section, `RebarSupervisor.cfg`
- **Why**: The real OneForAll race bug is two children getting the SAME PID, not a child missing a PID.

### 7. Supervisor tests only OneForOne strategy
- **What**: The cfg hardcodes `Strategy = "OneForOne"`. Add two more cfg files (`RebarSupervisor-OneForAll.cfg`, `RebarSupervisor-RestForOne.cfg`) or a single parameterized cfg.
- **Where**: New cfg files in `specs/tla/`
- **Why**: OneForAll and RestForOne restart strategies have distinct race conditions (the whole point of modeling them) but are never exercised by TLC.
