# Harden PR3: FFI Registry Integration

## Context
PR3 (`pr3-ffi-registry`) moved the name registry from a standalone `Mutex<HashMap>` in `RebarRuntime` (FFI layer) into the core `ProcessTable`. It also added `rebar_unregister` FFI + client wrappers. The code is functional but has several correctness, consistency, and coverage issues.

## Opportunity Details

### 1. TOCTOU race in `ProcessTable::register_name`
- **What**: `register_name` does `contains_key` then `insert` as separate DashMap operations. Under concurrent access, two callers could both pass the `contains_key` check and both insert, silently overwriting.
- **Where**: `crates/rebar-core/src/process/table.rs:102-111`
- **Why**: Correctness — the registry must enforce name uniqueness atomically.
- **Fix**: Use `DashMap::entry` API on `self.names` to do an atomic check-and-insert. Also verify PID exists first (this part is fine as-is since processes don't disappear mid-registration in practice, but we should keep the check).

### 2. `send_named` uses wrong error variant
- **What**: `Runtime::send_named` maps a missing name to `SendError::ProcessDead(ProcessId::new(0, 0))` — a synthetic PID `<0.0>` that is misleading. A name-not-found is semantically different from a dead process.
- **Where**: `crates/rebar-core/src/runtime.rs:152-154`
- **Why**: Callers can't distinguish "name not registered" from "process dead" — they get a fake PID. This should return a `RegistryError::NameNotFound` or the function signature should use a combined error type.
- **Fix**: Change `send_named` return type to `Result<(), SendNamedError>` with variants for both registry and send failures. Or simpler: add a `NameNotFound` variant to `SendError`.

### 3. `rebar_register` maps all errors to `REBAR_ERR_NOT_FOUND`
- **What**: When `rebar_register` fails because the name is already taken (`NameAlreadyRegistered`), it returns `REBAR_ERR_NOT_FOUND`, which is semantically wrong.
- **Where**: `crates/rebar-ffi/src/lib.rs:233-235`
- **Why**: FFI callers cannot distinguish "PID not found" from "name already taken". Need a new error code `REBAR_ERR_ALREADY_EXISTS`.
- **Fix**: Add `REBAR_ERR_ALREADY_EXISTS = -5`, match on specific `RegistryError` variants.

### 4. Missing `ProcessTable` registry unit tests
- **What**: No tests for `register_name`, `unregister_name`, `whereis`, or the cleanup-on-remove behavior.
- **Where**: `crates/rebar-core/src/process/table.rs` (test module)
- **Why**: The FFI tests cover the full stack but there are no isolated unit tests for the core registry logic.
- **Tests to add**: register+whereis, register duplicate name, register nonexistent PID, unregister, unregister not found, remove process cleans up names, whereis not found.

### 5. Missing `Runtime` registry unit tests
- **What**: No tests for `Runtime::register`, `unregister`, `whereis`, `send_named`.
- **Where**: `crates/rebar-core/src/runtime.rs` (test module)
- **Why**: Same — these are untested core APIs.

### 6. Idiomatic `copied()` instead of `.map(|r| *r)`
- **What**: `ProcessTable::whereis` uses `.map(|r| *r)` which is less idiomatic.
- **Where**: `crates/rebar-core/src/process/table.rs:129`
- **Fix**: Change to `.map(|r| *r.value())` — actually, since `DashMap::get` returns a `Ref`, `*r` derefs to the value via `Deref`. The current code works but `.map(|r| *r.value())` would be clearer. Actually `.map(|r| *r)` works fine here due to Deref. Minor style nit, skip if not worth it.
